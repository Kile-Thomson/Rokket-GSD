# Knowledge

<!-- Append-only. Read this at the start of every unit. -->

| # | When | Rule/Pattern | Detail |
|---|------|-------------|--------|
| 1 | 2026-03-12 | RPC mode drops factory functions | `ctx.ui.setWidget(key, factory)` is silently ignored in RPC mode. Only `setWidget(key, stringArray)` passes through. Any feature depending on TUI widget factories must be reimplemented extension-side. |
| 2 | 2026-03-13 | Windows spawn — never use shell:true with .cmd | Parse the .cmd wrapper and invoke `node <entry.js>` directly. Shell wrappers break on paths with spaces. |
| 3 | 2026-03-13 | VS Code env vars leak to child processes | ELECTRON_RUN_AS_NODE, NODE_OPTIONS, VSCODE_* must be stripped from env before spawning GSD. Otherwise grandchild processes (e.g. Next.js workers) can crash-loop. |
| 4 | 2026-03-14 | Dropbox conflicted copies | Dropbox creates `(Kile's conflicted copy)` files. Never read or edit these — always use the canonical filename. |
| 5 | 2026-03-14 | Buffer overflow → full reset, not truncation | JSONL protocol means truncating mid-line corrupts JSON. Full buffer reset preserves protocol integrity. |
| 6 | 2026-03-14 | Auto-mode state comes from setStatus events | `extension_ui_request` with `method: "setStatus"` and `statusKey: "gsd-auto"` carries the auto-mode state. Values: `"auto"`, `"next"`, `"paused"`, or `undefined` (stopped). |
| 7 | 2026-03-17 | get_state returns current model | When dynamic model routing switches models, `get_state` RPC returns the new model. No explicit "model_changed" event — must poll and compare. |
| 8 | 2026-03-17 | STATE.md milestone registry format | Parser expects `✅` for done and `⬜` for not-done. Other emoji (🔲, 🚧) won't parse correctly. |
| 9 | 2026-03-17 | Windows VSIX install from CLI | `code --install-extension` fails silently from bash because path spaces break the .cmd wrapper. Use `powershell.exe -Command "& 'C:\...\code.cmd' --install-extension 'path.vsix' --force"` instead. |
| 10 | 2026-03-19 | Release workflow ignores doc-only changes | `release.yml` path filters exclude README.md and CHANGELOG.md edits from triggering a release build. Prevents spurious empty releases when docs are updated directly on GitHub. |
| 11 | 2026-03-19 | stopReason "error" must be checked on message_end | Non-retryable agent errors (invalid API key, permission denied) were silently swallowed because message_end only checked for success. Always inspect stopReason on message_end events. |
| 12 | 2026-03-20 | VSIX install requires full VS Code restart | "Developer: Reload Window" does NOT reload a newly installed VSIX. A full VS Code quit + relaunch is required. Don't tell the user to reload — tell them to restart. |
| 13 | 2026-03-20 | .vscodeignore must exclude .gsd symlink itself | `.gsd/**` only matches contents, not the symlink entry. `vsce` secret scanner crashes with EISDIR on symlink-to-directory. Need both `.gsd` and `.gsd/**` in `.vscodeignore`. |
| 14 | 2026-03-20 | Stale echo detection misses post-steer job results | `detectStaleEcho` requires no user entry between assistant turns, but steer messages are user entries. Async job results after a steer don't get flagged as stale. Turn coalescing (`isContinuation` flag on `agent_start`) handles this at the extension host level instead. |
| 15 | 2026-03-20 | Inline code drifts from extracted modules | When code is extracted into a module but the inline copy isn't deleted, the two diverge silently. command-fallback.ts had 30+ subcommands vs. 9 inline; rpc-events.ts had turn-coalescing the inline lacked. Always wire extracted modules immediately and delete inline copies — "extract later, wire later" creates hidden drift. |
| 16 | 2026-03-20 | Context adapter getters must return fresh objects | Context adapters that capture mutable state (lastStatus, webviewView, session timers) must be getter properties, not cached fields. A cached adapter would snapshot stale state. The getter pattern `get ctx(): FooContext { return { ... } }` ensures each call sees current values. |
| 17 | 2026-03-20 | CSP style-src does not block JS .style.property | CSP `style-src` without `'unsafe-inline'` blocks HTML `style=""` attributes and `<style>` elements, but does NOT block JavaScript `.style.width = "50%"` assignments. Dynamic widths via JS are CSP-safe. Use `data-*` attributes in innerHTML templates + JS loop after rendering for dynamic values. |
| 18 | 2026-03-20 | gsd-hidden class is the only visibility toggle | Never use `el.style.display` in webview code. All show/hide uses `el.classList.add/remove('gsd-hidden')`. Visibility reads use `!el.classList.contains('gsd-hidden')`. Introducing style.display will trigger CSP violations for HTML attributes and silently bypass the class-based pattern for JS assignments. |
| 19 | 2026-03-20 | Async constructor init uses deferred .then() | VS Code webview provider constructors can't be async. Use `asyncInit().then(v => this.field = v)` for values only needed after construction. Don't await in constructor — it won't work. The field will be undefined until the promise resolves, so consumers must handle the transient state. |
| 20 | 2026-03-20 | fs.promises.exists doesn't exist | Node.js has no `fs.promises.exists`. Replace `existsSync(p) && readFileSync(p)` with `try { await fs.promises.readFile(p) } catch (e) { if (e.code === 'ENOENT') return default; throw e; }`. One syscall instead of two, fully async. |
| 21 | 2026-03-20 | Vitest fake timers don't advance real async I/O | When using `vi.useFakeTimers()`, any async function called from timer-driven code (like auto-progress poller) needs an explicit mock returning `mockResolvedValue`. Fake timers only advance timer scheduling — they don't resolve real filesystem promises. |
| 22 | 2026-03-20 | Wide context interfaces are a compile-time safety net | MessageDispatchContext has 15 properties + 3 nested contexts. This feels heavy, but TypeScript catches every mismatch at build time. The alternative (passing `this` directly) hides coupling. Accept the verbosity — it makes extraction boundaries explicit and refactoring errors impossible to miss. |
| 23 | 2026-03-20 | marked Lexer handles incomplete fences as normal code tokens | `marked.Lexer` produces a `code` token for both complete (```` ```js\n...\n``` ````) and incomplete (```` ```js\n... ````) fenced code blocks. No special detection of incomplete fences is needed — treating the last token as always in-progress covers the streaming case. |
| 24 | 2026-03-20 | Module-level Maps must be cleared on all reset paths | `incrementalState` Map in renderer.ts tracks per-segment frozen block counts. It must be cleared in `finalizeCurrentTurn()`, `resetStreamingState()`, and the stream-split path in `renderNewEntry()` — same places that clear `segmentElements`. Missing a reset path causes stale incremental state to corrupt rendering in the next turn. |
| 25 | 2026-03-20 | CSS import order in index.ts is the cascade | esbuild concatenates CSS imports in declaration order into one output file. The 16 CSS imports in webview/index.ts must be: tokens → base → components (in document order) → themes last. Inserting a new file in the wrong position creates silent visual regressions. Theme-internal custom properties (`--phosphor-*`, `--clarity-*`, `--forge-*`) must stay inside their theme file — don't hoist to tokens.css. |
| 26 | 2026-03-20 | Missing await on async calls causes silent test failures with fake timers | `auto-progress.ts` called `countPendingCaptures()` and `readParallelWorkers()` without `await`. With real timers this worked because the Promise was truthy and the `if (rawWorkers)` branch entered but threw internally (caught by try/catch). With fake timers the unresolved Promise caused `postMessage` to never fire, producing a confusing "0 calls" assertion failure. Always `await` async functions even when the return value is used in a truthiness check. |
| 27 | 2026-03-20 | Focus trap handler must be stored as module-level variable | Each overlay stores its `createFocusTrap()` handler in a module-level variable (e.g. `let focusTrapHandler: ((e: KeyboardEvent) => void) \| null = null`) so it can be cleanly removed via `removeEventListener` on close or re-render. Anonymous inline handlers can't be removed, causing handler accumulation on repeated open/close cycles. |
| 28 | 2026-03-20 | Changelog DOM lifecycle spans two modules | `keyboard.ts` creates the initial `#gsd-changelog` element (loading state), but `message-handler.ts` replaces its innerHTML when content arrives. Event listeners (focus trap, Escape handler, close button) attached in keyboard.ts are lost on replacement. Both modules must independently attach accessibility handlers to their respective DOM states. |
| 29 | 2026-03-20 | "Pre-existing" test failures deserve investigation, not acceptance | 2 auto-progress.test.ts failures were carried as "pre-existing" through all of M016 (4 slices) and M017 S01-S02. They were actually a real code bug (missing `await` + missing mock). The lesson: never normalize unexplained test failures. If a test failure can't be explained by a specific known cause, investigate it — it may signal a real code defect. |
| 30 | 2026-03-20 | Token layer completeness pays forward | Creating a comprehensive --gsd-* token layer (82 tokens) in T01 before CSS splitting and theme refactoring meant zero token additions were needed during the theme refactor pass. Front-loading the token inventory (spacing, radius, typography, colors, component-specific) reduces rework when downstream consumers adopt the tokens. |
| 31 | 2026-03-20 | Entry cap push-site enforcement is a convention, not a compile-time guarantee | `pruneOldEntries(messagesContainer)` must follow every `state.entries.push()` call. There are 8 call sites across 4 files (index.ts:4, message-handler.ts:3, renderer.ts:1). A new push site without the prune call silently breaks the cap. Consider adding a `grep -c "state.entries.push" \| grep -v pruneOldEntries` CI check to catch drift. |
| 32 | 2026-03-20 | Fork entryId is webview-local, not server-side (superseded by #44 — fork not supported) | Historical note only. The `entryId` passed to `client.fork(entryId)` is a sequential integer from the webview's `nextId()` counter, not the server-side RPC message ID. Fork is explicitly not supported per Rule 44. |
| 33 | 2026-03-20 | innerHTML wiring must re-run after overlay re-render | Retry overlay buttons (abortRetryBtn, forceRestartBtn, forceKillBtn) are wired via `getElementById` + `addEventListener` after the `innerHTML` update in `showRetryingIndicator()`. If the overlay is re-rendered (e.g., retry count updates), the wiring code must re-execute. Missing re-wiring makes buttons inert. Same fragility applies to any innerHTML-based UI that attaches handlers post-render. |
| 34 | 2026-03-20 | jsdom does not implement scrollIntoView | When testing webview components that call `el.scrollIntoView()` in jsdom, mock it globally: `Element.prototype.scrollIntoView = vi.fn()` in `beforeEach`. This affects slash-menu, session-history, model-picker, and any component that focuses/scrolls active items. Without this mock, tests crash with `TypeError: scrollIntoView is not a function`. |
| 35 | 2026-03-20 | Mock factory pattern for MessageDispatchContext | `createMockDispatchContext()` in message-dispatch.test.ts returns a `{ ctx, session, webview }` triple with all 15+ properties and 3 nested contexts mocked. Each test gets a fresh context — no shared mutable state. All transitive imports (vscode, watchdogs, command-fallback, file-ops, etc.) must be vi.mock'd at module level. This factory is the canonical pattern for testing any message-dispatch handler. |
| 36 | 2026-03-20 | createDeps() factory pattern for webview component tests | Webview modules that take a `deps` or initialization parameter use a `createDeps()` factory in tests that returns all required DOM elements and callbacks as mock objects. Combined with `vi.mock("../state")` to isolate state, this pattern covers model-picker, thinking-picker, session-history, slash-menu, and dashboard. |
| 37 | 2026-03-20 | Coverage target needs more files per percentage point as baseline rises | Going from 46.71% to 60% required 181 new tests across 12 files, not the 4 files originally estimated. The high-LOC zero-coverage modules (message-dispatch, dashboard) provided large jumps, but reaching the last few percent required testing many smaller modules. Future coverage targets should plan 10-15 test files per 10 percentage points above 50%. |
| 38 | 2026-03-20 | "Wired but unreachable" is worse than dead code | `set_auto_retry` has a working dispatch handler but no UI element posts the message. This is worse than a dead method — it creates the false impression the feature works. When wiring a dispatch handler, always verify the full path from UI element → postMessage → dispatch → RPC. If any link is missing, document it as a known gap or don't ship the handler. |
| 39 | 2026-03-20 | JSDoc all-or-nothing is cheaper than selective | Documenting all 69 exports across 3 files took marginally longer than doing the planned top-10. Once the JSDoc pattern (description + @param + @returns) was established for the first few, applying it to remaining exports was mechanical. Selective documentation requires a prioritization decision per export that costs more time than just writing the doc. |
| 40 | 2026-03-22 | User reports are already verified | When the user reports a problem, they have already done their checks. Never suggest they haven't updated, haven't restarted, or haven't done something they say they've done. If they say it's broken, investigate the code — don't question the report. |
| 41 | 2026-03-22 | Never push or release without explicit user confirmation | Every push to remote and every release trigger requires the user to say "yes, push it." Commits are fine — pushing is the boundary. The user tests locally before pushing. No exceptions, even if they gave blanket permission for a previous push in the same session. |
| 42 | 2026-03-22 | Trace all affected code paths before committing structural changes | When changing how a shared data structure works (e.g. clearing `segmentElements`, splitting `currentTurnElement`), trace every consumer of that structure and verify correctness before committing. Don't commit with known unknowns and plan to "address edge cases later." The investigation happens before the commit, not after the user catches it. |
| 43 | 2026-03-22 | User problems are problems — investigate, don't dismiss | When the user reports something isn't working, the response is investigation and a fix, not "just do something else." Suggesting workarounds like "start a fresh session" or "switch to a different one" instead of fixing the issue is unacceptable. Every problem reported is a problem to solve. |
| 44 | 2026-03-28 | GSD PI fork/getForkMessages — never implement | User decision: GSD PI's conversation fork feature (fork, get_fork_messages RPC commands) is explicitly not supported. Do not wire to UI, do not re-add on future GSD PI updates. The RPC methods are commented out in rpc-client.ts as dead code reference. |
| 45 | 2026-03-28 | GSD PI's VSCode extension features — not supported | We do not use GSD PI's own VSCode extension features (code lens, file decorations, activity feed, session tree, session forking). Our extension is the implementation. Skip these in future changelog audits. |
| 46 | 2026-03-28 | Dropbox Smart Sync breaks VSIX packaging | Dropbox can mark `dist/` files as cloud-only placeholders (mode flag `l`). `vsce package` silently skips them, producing a VSIX with missing webview JS/CSS → blank screen. Fix: delete `dist/` and rebuild before packaging. Also `dist/` must NOT be in `.gitignore` since `vsce` respects gitignore. |
| 47 | 2026-03-28 | exportHtml RPC is redundant | `client.exportHtml()` (server-side JSONL→HTML export) duplicates `handleExportHtml` in file-ops.ts (client-side DOM→HTML export). We use the client-side version. Don't re-add the server-side RPC method. |
| 48 | 2026-03-30 | OS pipe buffering causes burst delivery of text deltas | gsd-pi writes one JSONL line per token, but the OS pipe buffer accumulates multiple lines and delivers them in one `data` event. All deltas in a burst arrive synchronously, only the first schedules an rAF. Without a live Text node fast-path, all burst tokens render in one frame — giving chunky perceived output despite correct per-token data delivery. |
| 49 | 2026-03-30 | scrollHeight read forces synchronous layout — never in hot render path | `container.scrollTop = container.scrollHeight` in the per-delta path forced a layout reflow on every token, preventing paint batching and causing choppy rendering. Use ResizeObserver (deferred via rAF) for scroll — it fires after paint without blocking the compositor. |
| 50 | 2026-03-30 | innerHTML replacement resets CSS animations | Any `element.innerHTML = ...` call destroys and recreates all child DOM nodes, restarting animations from frame 0. For tool cards and subagent panels that contain spinners, use targeted DOM patching (update text content, swap classes, replace only changed child elements) instead of full innerHTML rebuild. |
| 51 | 2026-03-30 | data-* attribute mutation triggers style recalculation and resets animations | Writing to `element.dataset.foo` even with the same value triggers a style recalculation on the element and its subtree, restarting CSS animations on children. Always guard with `if (element.dataset.foo !== newValue)` before writing. |
| 52 | 2026-03-30 | prefers-reduced-motion blanket rule kills functional animations | Our base.css applies `animation-iteration-count: 1 !important` to all elements under `prefers-reduced-motion: reduce`. Windows "Animation effects" accessibility setting activates this. Functional loading indicators (spinners, thinking dots) must explicitly override with `animation-iteration-count: infinite !important` in a more specific reduced-motion rule. Use a gentler animation (opacity fade, slower speed) rather than suppressing iteration. |
| 53 | 2026-03-30 | async_subagent parallel mode requires tasks:[...] in a SINGLE call | The `async_subagent` tool supports `tasks: [{agent, task}, ...]` for parallel execution in one call. The old prompt guideline said "spawn multiple async_subagents" which caused the model to make 3 separate calls instead. The guideline AND the tool description must both be updated — the description is read first and carries more weight. Also update `~/.gsd/agent/extensions/async-subagent/index.js` directly since gsd-pi loads from there, not from the VSIX. A full VS Code restart is required after updating the live file. |
| 54 | 2026-03-30 | gsd-pi extension guidelines cached in memory — full restart required | gsd-pi loads extensions once at startup and caches them. Updating the extension JS on disk has no effect until gsd-pi restarts. A VS Code "Developer: Reload Window" is not sufficient — requires full VS Code quit + relaunch so the extension host respawns gsd-pi. |
